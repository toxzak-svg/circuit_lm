# Layered Context Architecture Experiment

A model-agnostic context management system with tiered storage and retrieval.

## Architecture Overview

### Tier 0 — Hot Working Set (Raw Tokens)
- Keeps last N tokens exactly (8K–32K)
- Used for precise local reasoning and code editing
- Direct access, zero latency

### Tier 1 — Structured Rolling State (Lossy but High Utility)
Instead of compressing old text into paragraph summaries, compress into state objects:
- **Symbols**: functions/classes touched, signatures, file paths
- **Constraints**: "must compile", "do not change API", "keep latency < X"
- **Decisions**: "we chose approach B because …"
- **Open loops**: TODOs, unresolved questions
- **Tests failing**: error traces distilled into key lines + file/line anchors

Represented as JSON/YAML for compactness.

### Tier 2 — Episodic Memory (Retrieval)
- Store chunks (code, logs, docs) in vector index + sparse index (BM25)
- Retrieve K most relevant chunks each turn
- Hybrid search for better coverage

### Tier 3 — Cold Archive (Full History)
- Everything else sits in cheap storage
- Only referenced via retrieval
- Complete audit trail

## Key Insight

**Most "need 128K" workloads are actually "need 10–20 relevant things from 128K."**

You don't just retrieve; you compile Tier 1 + retrieved chunks into a prompt program with strict formatting, and measure whether it preserves behavior.

## Installation

```bash
npm install
```

## Usage

```javascript
const { LayeredContext } = require('./src/orchestrator');

const context = new LayeredContext({
  tier0MaxTokens: 16384,
  tier2TopK: 10
});

// Add content
await context.add('code', 'function foo() { ... }', { file: 'src/app.js' });

// Query
const prompt = await context.compile('How does foo work?');
```
